# AI 对话知识沉淀系统任务拆分

## 1. 项目目标

每天结束时，系统扫描我和不同 AI 工具的对话记录，将其中值得长期保留的内容拆成原子化知识，进入个人知识库的待确认区。

第一版重点不是全自动收集所有来源，而是验证这件事：

- AI 提炼出的内容是否真的是我的观点、经验、素材或问题。
- 每条知识原子是否能回到原始对话证据。
- 每日沉淀结果是否值得我花几分钟确认。

## 3. 知识库承载方案

第一版采用：

- 本地 Markdown 作为知识本体和审核状态准来源。
- SQLite 作为标准化记录、导入记录、运行记录、知识索引和查询加速层。
- Obsidian 作为浏览、编辑和双链工具。
- `data/normalized/` 只作为可选调试导出或人工检查目录，不作为主数据源。
- 后续再按需要增加向量库索引。

暂不使用 Notion、飞书或外部数据库作为第一版知识主存储。Markdown 仍是知识本体，SQLite 只承载本地标准化记录、运行记录、导入记录和索引，避免一开始把系统做成云端知识平台。

如果后续作为可上线、可出售的桌面产品，需要按 `spec/产品化开发步骤.md` 推进。该文件会把安装包、跨平台、授权、更新、隐私、安全、日志和商业化能力纳入正式开发范围。

```text

```

目录说明：

- `raw/imports/` 存放原始导入记录，不直接当成知识。
- `data/normalized/` 可选存放标准化记录调试导出，不是主数据源。
- `data/daily_runs/` 存放每日运行结果和过程数据。
- `knowledge/inbox/` 存放待确认知识原子。
- `knowledge/approved/` 存放确认后的正式知识。
- `knowledge/rejected/` 存放明确不沉淀的候选内容。
- `knowledge/merged/` 存放已合并到其他知识原子的候选内容。
- `prompts/` 存放提炼、分类、去重相关提示词。
- `scripts/` 存放采集、标准化、提炼、写入脚本。
- `logs/` 存放运行日志。

## 5. 知识原子类型

第一版支持以下类型：

- 观点：我对某个问题的判断。
- 方法：我解决问题的一套做法。
- 决策：我在项目或创作中做出的选择。
- 经验：踩坑、复盘、规律识别。
- 素材：可用于公众号、文章、表达或案例的内容。
- 问题：值得继续追问的问题。
- 偏好：我反复表现出来的做事倾向。

## 6. 标准数据结构

### 6.1 标准化对话记录

```yaml
schema_version: string
record_id: string
source_app: codex | cursor | deepseek | doubao | workbuddy
source_type: local_app | web_export | manual_export
conversation_id: string
parent_conversation_id: string
turn_index: number
message_index_start: number
message_index_end: number
message_time: string
project: string
topic: string
user_message: string
ai_message: string
raw_path: string
raw_source: string
sensitivity: personal | private | confidential
can_enter_personal_kb: true
created_at: string
updated_at: string
```

### 6.2 知识原子

```yaml
schema_version: string
atom_id: string
title: string
type: 观点 | 方法 | 决策 | 经验 | 素材 | 问题 | 偏好
content: string
source_app: codex | cursor | deepseek | doubao | workbuddy
source_record_ids:
  - string
source_raw_paths:
  - string
project: string
tags:
  - string
sensitivity: personal | private | confidential
review_status: pending | approved | rejected | merged
created_at: string
updated_at: string
evidence: string
```

Markdown 写入模板以 `spec/知识原子Markdown模板.md` 为准。

### 6.3 版本和迁移约束

- 所有核心数据结构必须包含 `schema_version`。
- 所有核心数据结构必须包含 `created_at` 和 `updated_at`。
- `record_id`、`atom_id`、`run_id` 必须稳定，不随标题或内容轻微修改而变化。
- Markdown 是知识正文和审核状态的准来源。
- SQLite 是标准化记录、导入记录、运行记录、知识索引和查询加速层。
- 标准化记录的主数据源是 SQLite，`data/normalized/` 只用于可选调试导出或人工检查。
- 每日沉淀流程从 SQLite 读取标准化记录，不从 `data/normalized/` 读取主流程数据。
- SQLite 中的知识索引必须能从 Markdown 重建；标准化记录可通过原始归档重新标准化生成。

### 6.4 标准化记录粒度

一条 `NormalizedRecord` 表示一个用户请求及其对应 AI 回复的单轮记录。

多轮会话处理规则：

- `parent_conversation_id` 表示原始多轮会话。
- `record_id` 表示拆分后的单轮记录。
- `turn_index` 表示该轮在会话中的顺序。
- `message_index_start` 和 `message_index_end` 表示该轮对应的原始消息范围。
- 工具调用、命令输出、文件修改摘要不直接塞进 `user_message` 或 `ai_message`，后续应进入上下文摘要或附件引用。

追溯字段规则：

- `raw_path` 是导入器识别到的 vault-relative path，标准化输出必须存在。
- vault 根目录由本地设置保存，运行时解析为绝对路径。
- `raw_source` 是可选的人类可读来源说明。
- 知识原子的追溯优先使用 `source_record_ids` 和 `source_raw_paths`。

## 7. 原子化规则

- 一条知识原子只表达一个核心观点、经验、方法或问题。
- 不能只保留 AI 的泛泛建议，必须能对应到我提出的问题、判断、选择或表达。
- 每条知识原子必须保留来源平台和原始记录引用。
- 项目临时执行细节不进入正式知识库，除非能沉淀为可复用经验。
- 公众号素材可以进入候选区，但需要标记为素材，不伪装成观点。
- 同一想法反复出现时，不新增重复卡片，应合并到旧卡片或标记为补充证据。

## 9. 第一版验收标准

- 能从 Codex 一个来源导入原始记录。
- 能将原始记录转换为统一结构。
- 能生成一份每日沉淀文件。
- 能生成待确认知识原子。
- 每条知识原子都能追溯到原始对话。
- 不把公司项目和敏感内容写入个人正式知识库。
- 不自动写入正式库，必须先进入待确认区。

## 10. 暂不做事项

- 不做网页自动登录和自动抓取。
- 不接入公司项目。
- 不做多端同步。
- 不做复杂 UI。
- 不做向量数据库。
- 不做自动发布公众号。
- 不直接把候选知识写入正式知识库。
