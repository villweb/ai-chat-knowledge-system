### 问题 2：SQLite 存储实现与 `StorageProvider` 契约关系不清楚

问题：

`StorageProvider` 中的 `saveNormalizedRecords` 和 `findNormalizedRecords` 是异步接口，返回 `Promise`；当前 `SQLiteNormalizedRecordStore` 提供的是同步方法，并且没有声明实现 `StorageProvider` 或某个明确的子接口。

风险：

如果后续 Core Engine 依赖 `StorageProvider`，当前 SQLite 存储类无法直接作为实现注入。执行者可能再写一层适配器，也可能绕过 `StorageProvider` 直接调用 SQLite 类，导致 Core Engine 写入边界变得分裂。

依据：

- `app/storage/storage-provider.ts` 定义 `saveNormalizedRecords(records): Promise<void>` 和 `findNormalizedRecords(...): Promise<NormalizedRecord[]>`。
- `app/storage/sqlite-normalized-record-store.ts` 定义同步方法 `saveNormalizedRecords(...): void` 和 `findNormalizedRecords(...): NormalizedRecord[]`。
- P0/P1 规划强调 Core Engine 是唯一写入入口，StorageProvider 是核心存储边界。

建议动作：

明确关系。可以选择：

- 让 `SQLiteNormalizedRecordStore` 只作为底层同步 DAO，并新增一个实现 `StorageProvider` 或 `Pick<StorageProvider, ...>` 的适配层。
- 或者调整当前类的方法签名为异步并声明实现明确接口。

不要让 Core Engine 同时依赖 `StorageProvider` 和底层 SQLite DAO。

产品化规划把隐私、安全和合规作为 P7，但 P2 就已经开始 AI 提炼、隐私和敏感度判断、证据摘录。只要调用云端 AI，就可能涉及原始对话内容外发。

风险：

如果等到 P7 才补隐私策略，P1/P2 的数据流可能已经默认允许上传原始聊天记录，违背“本地优先”和“授权采集”的产品第一原则。

依据：

- `spec/产品化开发步骤.md` 第 1 节产品第一原则包含本地优先、授权采集、人工确认和可追溯。
- `spec/产品化开发步骤.md` P2 包含 AI 提炼、隐私和敏感度判断、证据摘录。
- `spec/产品化开发步骤.md` P7 才系统性处理隐私分级、敏感内容本地识别、API Key 安全保存和隐私政策。

建议动作：

把最低限度隐私约束前置到 P0/P1/P2：明确默认不上传原始记录、AI 调用前必须经过用户授权、公司项目和敏感客户信息默认阻断、日志不得写入 API Key 或大段原文。

### 问题 5：首个 MVP 到底跑通一个来源还是三个来源不够清晰

问题：

`spec/任务拆分.md` 第一版验收标准是“能从至少一个来源导入原始记录”，而 `spec/产品化开发步骤.md` P4 的验收标准是“首批三个来源至少支持一种稳定导入方式”。首个可开发版本又写“Codex、Cursor、DeepSeek 至少一个来源跑通”，表达容易被理解成两种不同范围。

风险：

执行者可能在 MVP 阶段同时实现三个连接器，导致核心闭环被连接器细节拖慢；也可能只做一个来源，却在 P4 验收时被误判为没完成。

依据：

- `spec/任务拆分.md` 第 9 节要求至少一个来源导入。
- `spec/产品化开发步骤.md` 第 5 节要求 Codex、Cursor、DeepSeek 至少一个来源跑通。
- `spec/产品化开发步骤.md` P4 验收标准要求首批三个来源至少支持一种稳定导入方式。

建议动作：

明确阶段口径：本地 MVP 只要求三选一跑通完整闭环，建议优先 Codex；P4 再要求 Codex、Cursor、DeepSeek 三个来源分别具备稳定导入方式。

### 问题 6：产品级目录规划还没有落地，但不应在 P0 前盲目创建

问题：

实际项目目录目前仍是概念验证骨架，只有 `raw/`、`data/`、`knowledge/`、`prompts/`、`scripts/`、`logs/`、`spec/`，还没有 `app/` 和 `build/`。这本身不是问题，但新规划已经把产品级目录列为下一步。

风险：

如果没有先完成技术栈和模块边界决策，直接创建 `app/desktop/`、`app/core/`、`app/storage/` 等目录，可能很快变成空目录堆叠，后续真实框架初始化时还要搬迁。

依据：

- `spec/产品化开发步骤.md` 第 3 节建议产品化目录演进为 `app/desktop/`、`app/core/`、`app/connectors/`、`app/storage/`、`app/workflow/`、`app/schemas/`、`app/services/` 和 `build/`。
- 当前项目实际目录中尚无 `app/` 和 `build/`。

建议动作：

先完成 P0 技术决策，再一次性创建与技术栈匹配的最小目录。目录不宜只为了“看起来产品化”而提前铺满。

### 问题 7：缺少版本化数据结构和迁移策略

问题：

新规划已经进入可安装、可更新、可长期维护的产品方向，但目前标准数据结构没有 `schema_version`、迁移策略、唯一 ID 生成规则和兼容策略。

风险：

后续一旦用户产生真实数据，字段调整会影响 SQLite 表、Markdown front matter、每日运行记录和历史知识卡片，升级时容易出现旧数据无法读取。

依据：

- `spec/任务拆分.md` 第 6 节定义了 `NormalizedRecord` 和知识原子字段。
- `spec/产品化开发步骤.md` P1 要定义 `NormalizedRecord`、`KnowledgeAtom`、`DailyRun`，P8 要支持安装、升级和卸载。

建议动作：

在 P1 数据结构定义中加入 `schema_version`、`created_at`、`updated_at`、稳定 ID 规则、迁移入口和向后兼容原则。即使第一版不实现完整迁移，也要先保留版本字段。

### 问题 8：商业化和自动更新被列入架构，容易误伤 MVP 范围

问题：

产品架构里包含 `License and Update`，P8/P9 也详细规划了自动更新和商业化能力，但第 5 节又明确首个可开发版本暂缓商业授权和自动更新。

风险：

后续 AI 可能因为看到架构图里的 `License and Update`，提前实现授权、购买入口、自动更新等非 MVP 能力，挤占核心闭环开发时间。

依据：

- `spec/产品化开发步骤.md` 第 2.2 节桌面端模块包含 `License and Update`。
- `spec/产品化开发步骤.md` 第 5 节暂缓包含商业授权和自动更新。
- `spec/产品化开发步骤.md` P8/P9 才是跨平台发布和商业化能力。

建议动作：

在 P0 第一版产品需求清单中明确：MVP 阶段只保留接口或占位，不实现授权、购买、自动更新和官网相关能力。

### 问题 9：还缺少每次执行后的执行日志机制

问题：

依据：

建议动作：

建议在开始真正编码前创建 `spec/执行日志.md`，每次执行按倒叙记录：日期、执行目标、修改文件、完成项、未完成项、偏离 spec 的地方和下一步。

## 更好的监督方式建议

建议采用“三文档分工”：

- `spec/任务拆分.md`：只放目标、范围、任务拆分和验收标准，保持稳定。
- `spec/监督问题记录.md`：记录执行偏移、风险和挑刺建议，最新在上。
- `spec/执行日志.md`：记录每次实际做了什么、修改了哪些文件、是否完成验收。

这样后续 AI 可以先读 `任务拆分.md` 理解目标，再读 `监督问题记录.md` 避免踩坑，最后读 `执行日志.md` 判断当前进度。
