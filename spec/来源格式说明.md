# 来源格式说明

本文档记录各 AI 来源的获取方式、原始格式差异和首版处理策略。后续编写标准化逻辑时，以本文档作为输入依据。

## 1. 通用手动导入格式

首版先允许用户手动放入以下格式：

- Markdown
- TXT
- JSON

真正的解析支持在 P1-12 实现。阶段 1 只表示入口目录、示例和格式约定已经存在。

推荐每份手动导入记录至少包含：

```yaml
source_app: codex | cursor | deepseek | doubao | workbuddy
source_type: manual_export | local_app | web_export
conversation_id: string
message_time: string
project: string
topic: string
user_message: string
ai_message: string
raw_path: string
raw_source: string
```

字段职责：

- `raw_path` 是导入器识别到的 vault-relative path，标准化输出必须存在。
- `raw_source` 是可选的人类可读来源说明，可以为空。
- 如果导入文件本身声明了 `raw_path`，导入器仍以实际读取路径为准。
- vault 根目录保存在本地设置中，运行时由 `vault_root + raw_path` 解析到真实文件。
- AI 请求中永远不发送原始 `raw_path`。

如果用户导入的是非结构化 Markdown 或 TXT，标准化阶段需要尽量识别用户消息、AI 回复、时间、主题和来源；无法识别的字段标记为 `unknown`，不得凭空补造。

## 2. JSON 输入形态

JSON 允许两种形态。

### 2.1 简单单轮形态

```json
{
  "source_app": "codex",
  "source_type": "manual_export",
  "conversation_id": "example-conversation",
  "message_time": "2026-06-18T10:00:00+08:00",
  "project": "AI对话知识沉淀系统",
  "topic": "示例主题",
  "raw_path": "raw/imports/codex/example.json",
  "raw_source": "手动导入",
  "user_message": "用户消息",
  "ai_message": "AI 回复"
}
```

映射规则：

- 生成一条 `NormalizedRecord`。
- `turn_index` 为 `0`。
- `message_index_start` 为 `0`。
- `message_index_end` 为 `1`。

### 2.2 多消息形态

```json
{
  "source_app": "codex",
  "source_type": "manual_export",
  "conversation_id": "example-conversation",
  "messages": [
    { "role": "user", "content": "用户消息" },
    { "role": "assistant", "content": "AI 回复" }
  ]
}
```

映射规则：

- 按 `user` 消息及其后续最近一条 `assistant` 消息组成一条 `NormalizedRecord`。
- 每组记录按顺序递增 `turn_index`。
- `message_index_start` 和 `message_index_end` 使用原始 `messages` 数组下标。
- 如果出现连续用户消息，导入器应生成解析错误并进入人工检查。
- 如果缺少对应 AI 回复，导入器应生成未完成记录并进入人工检查。

## 3. Codex

首版策略：P4 已支持手动路径导入，入口为 `raw/imports/codex`。

获取方式：

- 用户手动导出或复制对话为 Markdown、TXT 或 JSON。
- 用户手动选择本地记录文件或项目记录目录。

需要记录的字段：

- 对话时间。
- 所属项目。
- 用户请求。
- AI 回复。
- 修改文件或任务上下文。
- 原始文件路径。

已知限制：

- 不同 Codex 环境的本地记录路径和格式可能不同。
- 项目修改类对话可能混有大量命令输出和过程日志，需要在标准化后过滤低价值片段。
- 连接器只读取 `raw/imports/codex` 下用户放入或选择的 Markdown、TXT、JSON 文件，不读取浏览器会话或系统任意目录。

## 4. Cursor

首版策略：P4 已支持手动路径导入和本地记录副本识别，入口为 `raw/imports/cursor`。

获取方式：

- 用户手动导出或复制聊天记录。
- 用户手动选择 Cursor 相关本地记录文件。

需要记录的字段：

- 对话时间。
- 工作区或项目名称。
- 用户消息。
- AI 回复。
- 关联文件或代码片段。
- 原始文件路径。

已知限制：

- 本地存储格式可能随 Cursor 版本变化。
- 代码问答碎片较多，需要区分项目决策、代码事实和个人偏好。
- 当前不直接读取 Cursor 数据库原件，不自动遍历工作区代码；推荐 JSON 使用 `source_app: cursor` 和 `source_type: local_app`。

## 5. DeepSeek

首版策略：P4 已支持网页导出文件导入，入口为 `raw/imports/deepseek`。

获取方式：

- 网页端手动复制或导出对话。
- 用户保存为 Markdown、TXT 或 JSON 后导入。

需要记录的字段：

- 对话时间。
- 会话标题。
- 用户问题。
- AI 回复。
- 原始导出文件路径。

已知限制：

- 不做自动网页登录和自动抓取。
- 网页复制内容可能缺失精确时间，需要允许 `message_time` 为 `unknown`。
- 推荐 JSON 使用 `source_app: deepseek` 和 `source_type: web_export`。

## 6. 豆包

首版策略：只预留目录和连接器入口，不进入 MVP。

获取方式：

- 后续再调研手动导出、复制或本地记录方式。

已知限制：

- 内容可能更偏灵感和素材，需要更强的低价值内容过滤。
- 不做自动登录和绕过平台权限的采集。

## 7. Workbuddy

首版策略：只预留目录和连接器入口，不进入 MVP。

获取方式：

- 后续在确认公司项目和权限边界后再设计。

已知限制：

- 可能涉及公司项目和敏感信息。
- 默认不接入个人知识库。
- 接入前必须先补充公司项目隔离库、权限和隐私 spec。

## 8. 标准化注意事项

- 每条标准化记录必须保留 `raw_path`。
- `raw_source` 只作为可选说明，不用于替代追溯路径。
- `raw_path` 持久化为 vault-relative path。
- 无法确认来源权限的数据不得进入标准化流程。
- 无法解析的内容不得丢弃，应该记录错误并进入人工检查。
- 连接器只负责读取和初步解析，不负责写入正式知识库。
- 所有写入必须经过 Core Engine。

## 9. P4 连接器注册状态

| 来源 | 状态 | 默认启用 | 导入目录 | 支持格式 | 读取范围 |
|---|---|---:|---|---|---|
| Codex | 可用 | 是 | `raw/imports/codex` | Markdown、TXT、JSON | 用户选择或放入该目录的本地文件 |
| Cursor | 可用 | 是 | `raw/imports/cursor` | Markdown、TXT、JSON | 用户选择或放入该目录的 Cursor 导出或本地记录副本 |
| DeepSeek | 可用 | 是 | `raw/imports/deepseek` | Markdown、TXT、JSON | 用户手动保存的网页导出、复制内容或 JSON 文件 |
| 豆包 | 预留 | 否 | `raw/imports/doubao` | Markdown、TXT、JSON | P4 不读取，仅显示配置入口 |
| Workbuddy | 预留 | 否 | `raw/imports/workbuddy` | Markdown、TXT、JSON | P4 不读取；涉及公司项目前必须补权限和隔离策略 |

P4 明确不做自动网页登录、浏览器 Cookie 读取、网页自动抓取、系统全盘扫描或公司项目接入。
