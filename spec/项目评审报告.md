## 2. 可行性评审

### 2.1 技术路线可行，但不是主要风险

Electron + React + TypeScript Core + SQLite + Markdown 的组合成熟、可落地，方向没有问题。需要登记的技术风险只有一个偏后期的：`better-sqlite3` 是**原生模块**，Electron 打包必须做 `electron-rebuild` 或跨平台预编译，否则 P8 跨平台安装包会在用户机器上加载失败。建议现在就把这条写进 P8 的已知风险。

### 2.2 真正的可行性风险：范围与产出节奏严重不匹配

- 规划覆盖 P0–P10：本地核心、AI 提炼、桌面 UI、连接器、定时任务、知识库体验、隐私合规、跨平台签名打包、商业授权、上线质检。这是一个**多人月级别的完整商业产品**蓝图。
- 当前实际产出：约 1270 行代码，且其中**大部分是类型定义和孤立工具函数**；多轮迭代后 P1 仍未形成可运行的端到端闭环（见第 3、4 节），P2（AI 提炼）零进展。
- **项目最核心、风险最高的假设——"AI 自动提炼出的内容是否真的是我的观点/经验/素材，是否值得每天花几分钟确认"——恰恰落在尚未启动的 P2。** 目前所有已完成工作都还没触及这个假设。

可行性结论：**技术可行，但按当前节奏走完全程的可行性偏低，且把大量精力投在了验证价值之前的脚手架上。** 建议重排优先级（见第 5 节），先用最小代价验证核心价值假设，再决定是否投入桌面化/打包/商业化。

### 2.3 过程开销偏重

监督文档（1205 行）+ spec 文档体量已明显超过实现代码（~1270 行且多为类型）。规划与监督本身是优点，但"规划/监督占比 ≫ 可运行实现产出"是一个需要警惕的信号：流程在空转，价值未被验证。

## 4. 问题清单（按严重度）

### 🔴 严重（阻断 P1 验收 / 影响后续所有阶段）

**问题 1：P1 声称的"闭环"实际不可端到端运行，缺少编排层与入口**

- `app/core/core-engine.ts` 只有 `CoreEngineDependencies` 和 `CoreEngine` 两个 **interface**，**没有任何实现**，没有把"连接器列举→读取→归档→标准化→写 SQLite→写 Markdown"串起来的编排函数。
- 全仓库**没有任何类实现完整 `StorageProvider`**（`grep "implements StorageProvider"` 无结果），也**没有 `new CoreEngine` / `main` / CLI 入口**，`scripts/` 仅有 `.gitkeep`，`package.json` 只有 `typecheck` 一个脚本。
- 后果：各模块是孤立可用的单元，但"跑一次完整闭环"必须靠仓库外的临时脚本，无法被他人/未来的自己复现，P1 验收无法稳定证明。
- 建议：新增一个最小编排（`CoreEngine` 实现或 `runImport()` 函数）+ 一个可执行入口（如 `scripts/run-import.ts` 或 `npm run import -- <vault>`），把现有零件接通，让 P1 验收变成一条命令。

**问题 2：没有任何自动化测试，"冒烟验证通过"不可复现**

- `执行日志.md` 每轮都写"已用临时 vault 冒烟验证""已验证脱敏结果"，但**验证脚本/测试均未提交**，`*.test.ts` / `*.spec.ts` 数量为 0。
- 后果：spec 里多条验收（"重复运行不会重复写入""SQLite 可从 Markdown 与原始归档重建""日志脱敏"）没有任何回归保护，一次重构就可能静默破坏，且无法向后续接手者证明它们曾经成立。
- 建议：至少为 `normalizeManualImport`（JSON 单轮/多消息/Markdown/TXT、连续 user 报错、缺失 assistant 报错）、`sanitizeLogMessage`（sk- 脱敏）、`toSafeTitle`、SQLite upsert 幂等性补单元测试，并加入 `npm test`。

**问题 3："原始记录归档"实为自我覆盖，且校验值未持久化**

- `LocalRawArchive.archive` 把 `document.content` 写回 `document.raw_path`，而 `raw_path` 正是导入时读取的**同一个文件**（`ManualImportConnector` 从 `raw/imports/<app>/` 读取并把该路径作为 `raw_path`）。结果是"用相同内容覆盖源文件"，**并没有产生独立、不可变的证据副本**，与"`raw/` 作为证据归档"的设计目标不符。
- 计算出的 `checksum` 计算后**没有写入任何地方**：`normalized_records` 表无 checksum 字段，也没有归档登记表。P0 第 8 节要求"标准化记录…同时保留原始路径和校验信息"未落地。
- 建议：归档应写到与导入目录分离的不可变位置（如 `raw/archive/<app>/<record或hash>.<ext>`），并把 `checksum`、`archived_path` 持久化（新增表或在标准化记录中加列），追溯才真正成立。

### 🟠 重要（契约分裂 / 与红线方向相反 / 验收未真正达成）

**问题 4：`StorageProvider` 契约与实现命名、分工不一致，部分方法完全未实现**

- 接口定义 8 个方法：`ensureWorkspace / archiveRawDocument / saveNormalizedRecords / findNormalizedRecords / saveDailyRun / writeKnowledgeAtom / mirrorKnowledgeAtomIndex / findKnowledgeAtom`。
- 实际：`saveNormalizedRecords`/`findNormalizedRecords` 在 `SQLiteNormalizedRecordStore`；`writeKnowledgeAtom` 在 `KnowledgeAtomMarkdownWriter`；归档逻辑叫 `LocalRawArchive.archive`（**与接口名 `archiveRawDocument` 对不上**）；目录确保逻辑是独立函数 `ensureWorkspaceDirectories`（与接口名 `ensureWorkspace` 对不上）。
- **`saveDailyRun`、`mirrorKnowledgeAtomIndex`、`findKnowledgeAtom` 三个方法没有任何实现**；`DailyRun` 结构定义齐全但没有任何写入/生成路径。
- 这正是上一轮监督"问题 2"担心的"Core Engine 写入边界分裂"——现在已经发生。建议：要么提供一个聚合类真正 `implements StorageProvider` 并委托给各子实现，要么收敛接口到当前真实能力，避免接口与实现长期对不上。

**问题 5：隐私默认姿态与 spec 红线方向相反（默认放行而非默认阻断）**

- spec 反复强调"无法确认权限/来源的数据**默认阻断**"。但 `normalize-manual-import.ts` 对 JSON、TXT **硬编码 `sensitivity: "personal"`**，且 `can_enter_personal_kb = sensitivity !== "confidential"` → 实际是**默认放行进入个人知识库**。Markdown 走 front matter，但缺省同样回落到 `personal`。
- P1 不实现隐私分类器是可接受的（属 P2），但"默认放行"是与底线**相反**方向的占位，极易在后续被遗忘。建议：把无法判定来源的默认值改为更保守（如默认 `private` 且 `can_enter_personal_kb=false`，需显式确认才放行），或至少在代码处留显式 `TODO(P2 privacy)` 并在 spec 记一笔。

**问题 6："SQLite 可从 Markdown/原始归档重建"与"迁移入口"只是类型，未达可执行验收**

- `migration.ts` 仅有类型；`schema_migrations` 表建了但**从不写入**；`SCHEMA_VERSION` 一旦变更没有任何迁移执行器。
- 没有任何"从 Markdown 重建索引"或"从原始归档重新标准化"的代码。该项作为 P1 验收标准目前**仅停留在文档承诺**。建议：要么补一个最小可运行的 rebuild 命令，要么在文档里把该验收明确降级为"P1 仅预留类型，重建能力在 Px 实现"，避免验收名实不符。

### 🟡 次要（工程质量 / 一致性）

**问题 7：`resolveVaultPath` 在 6 处逐字复制**——connector、workspace、raw-archive、sqlite-store、markdown-writer、logger 各抄一份相同的路径越界校验。这是关键安全逻辑，分散后改一处极易漏改。建议抽到 `app/storage/paths.ts` 统一导出。

**问题 8：导入端与写出端的 YAML 处理不对称**——写 Markdown 用 `yaml` 库（支持数组/嵌套），但读 Markdown front matter 用的是 `parseFrontMatter` 自研逐行 `key:value` 解析器，**不支持数组、多行、嵌套值**。稍复杂的 YAML 会报错或静默丢字段。建议导入端也改用 `yaml` 解析，读写对称。

**问题 9：`source_app` 来源职责重叠**——`ManualImportConnector` 构造时已传入 `source_app`，`listCandidates` 又按 `/imports/<app>/` 过滤，`inferSourceApp` 还从路径再推断一次，三处职责重叠；且 `inferSourceApp` 对 `doubao`/`workbuddy` 也无差别识别，与"默认不接入豆包/Workbuddy"的范围约束存在张力。建议收敛为单一来源判定。

**问题 10：仓库边界混乱**——`ai-chat-knowledge-system` 与个人创作内容（`wiki/`、`封面图/`、`xiaoxiao-renxing-writing.skill`、`选题库` 等）共处同一上层目录与同一 Obsidian/Workspace 空间。虽然各自有独立 git，但容易在导入测试时把个人创作数据误带进来。建议明确物理隔离或在 spec 注明边界。

## 5. 下一步建议（按优先级）

1. **先把 P1 真正闭环跑通（最高优先）**：补一个 `CoreEngine`/`runImport` 编排 + 一条可执行命令，让"Codex 示例 → 标准化 → SQLite → 待确认区 Markdown"一键复现。这是把"模块级完成"变成"阶段级完成"的关键一步（对应问题 1、4）。
2. **补最小测试套件**，锁住已声称的验收行为，给后续重构兜底（对应问题 2）。
3. **修正归档语义与校验持久化**（对应问题 3），让"可追溯"名实相符。
4. **尽快做一次 P2 的最小真实提炼实验**：哪怕只接一个模型、只跑示例记录、只产出 1～3 张候选卡片，也要**尽早验证"AI 提炼是否真有价值"这个核心假设**。建议把它提到桌面 UI（P3）和打包（P8）之前。当前最大的风险不是技术，而是把大量精力投在了价值验证之前。
5. **调整隐私默认值方向**，使其与"默认阻断"红线一致（对应问题 5）。
6. 工程清理（问题 7–10）可在上述之后批量处理。

## 6. 一句话总结

规划与纪律是这个项目的最大资产，可惜目前的产出仍停留在"高质量的脚手架与类型契约"，尚未跑通一次可复现的端到端闭环，也尚未触碰"AI 提炼是否真有价值"这一核心假设。建议把重心从"继续完善规划/补齐 P1 模块"，转向"用最小代价让闭环可运行、让价值假设被验证"。

### 7.2 新增问题

**🔴 问题 11：唯一可运行入口在干净环境下直接崩溃（原生模块 ABI 不匹配）**

- 直接运行 `npm run p1:codex-import` 报 `ERR_DLOPEN_FAILED`：`better_sqlite3.node was compiled against NODE_MODULE_VERSION 131 ... requires 127`。必须手动执行 `npm rebuild better-sqlite3` 后才能跑通。
- `package.json` 无 `postinstall`/`electron-rebuild`，没有任何自动重建钩子。后果：旧报告 2.1 把 `better-sqlite3` 原生模块风险登记为"P8 打包时才需关注"，但**该风险此刻在开发环境就已发生**——"P1 一键复现"在 `npm install` 后的首次运行即失败。
- 建议：加 `postinstall` 自动 `npm rebuild`（或锁定 Node ABI / 改用 `node:sqlite` 等纯托管方案），让"clone → install → 一条命令"真正可复现。

**🔴 问题 12：闭环止于 SQLite，从不写"待确认区"Markdown（写卡步骤缺失，Writer 是死代码）**

- 实测：跑完导入后 `knowledge/inbox/` 仍只有 `.gitkeep`，**没有产生任何卡片**。
- 全仓库无任何代码调用 `KnowledgeAtomMarkdownWriter.writeKnowledgeAtom`——它是**死代码**。runner 只做"归档 + 写 SQLite"，"导入→标准化→入库→**写卡**"实际是"导入→标准化→入库"。
- 后果：人最终要确认的可读产物（待确认卡片）根本没被生成；P1 设想里"不打开桌面端也能产出待确认内容"这一价值出口缺位。
- 建议：明确 P1 是否应产出 inbox Markdown。若应产出，则把 Writer 接入 runner；若刻意推迟到 P2/AI，则在 spec 注明并把 Writer 标记为预留，避免"已实现却从不调用"的死代码长期存在。

### 8.2 新发现问题

**🔴 问题 13：从原始归档重建出的 `record_id`（及 `atom_id`、`conversation_id`）与原始导入不一致，"可重建"名实不符**

- 根因：`createRecordId` 把 `document.raw_path` 纳入哈希种子。导入时 `raw_path = raw/imports/...`，而重建时读取的是 `raw/archive/<checksum>...`，**同一份内容因路径不同而生成不同稳定 ID**。
- 实测对比（同一份 Codex 示例）：
  - 导入：`rec_bad2d3a05a58a697b76353eb`（md）、`rec_bdc4b30c7e03ed2291ad0577`（json）
  - 重建：`rec_0a71be2dc338dc1d872111dd`（md）、`rec_bb730cea849801110cf5c4d2`（json）
- 后果：① 违反 P0"稳定 ID = 内容哈希"契约（`raw_path` 是位置不是内容）；② 重建进**已有库**会按 record_id upsert，新 ID 当成新行 → **产生重复记录**而非幂等对账；③ `atom_id` 派生自 `record_id`，重建后知识原子的 ID、文件名、`source_record_ids` 全部漂移，索引与卡片链接断裂；④ TXT 的 `conversation_id = hash(raw_path)` 同样漂移。
- 建议：稳定 ID 的种子只用**内容相关**字段（如 `source_app + conversation_id + turn_index + user_message + ai_message`），剔除 `raw_path`；或重建时把归档路径回映射为导入路径口径。修复后补一条"重建后 record_id/atom_id 与导入完全一致"的断言。

**🟠 问题 14：重建单测只断言数量、不校验 ID 等价，掩盖了问题 13**

- `tests/p1-core.test.ts` 对 `rebuildLocalIndexes` 仅断言 `archived_file_count/normalized_record_count/knowledge_atom_count` 均为 2，**未断言重建出的 record_id/atom_id 等于原始导入**；且重建运行在已填充的库上会留下重复行，而测试不检查库内总行数。这正是问题 13 长期潜伏却"测试通过"的原因。
- 建议：补充"重建 ID 与导入 ID 集合相等""重建后库内行数不翻倍"两条断言。

**🟠 问题 15：`raw_path` 列在两条写入路径下语义漂移**

- 导入路径写入 `raw_path = raw/imports/...`，重建路径写入 `raw_path = raw/archive/...`。同一列在不同来源下含义不一致，下游若依赖该列回溯会得到不同结果。建议统一为单一口径（建议恒为导入相对路径，归档路径单列 `raw_archive_path` 已存在）。

**🟡 问题 8（仍未解，且更分裂）**：front matter 读写仍不对称——导入端 `normalize-manual-import.ts` 仍用自研逐行 `parseFrontMatter`（不支持数组/嵌套/多行），而 `rebuild-local-indexes.ts` 与 writer 已改用 `yaml` 库。同一仓库现存**两套** front matter 解析策略，复杂 YAML 在导入端会报错或丢字段。建议导入端统一改用 `yaml`。

### 8.3 观察（非缺陷，但需在 spec 标注）

- P1 待确认卡片是**固定占位文案**（`buildP1PendingKnowledgeAtom` 对每条记录生成相同模板正文），符合 P1"只验证写入待确认区"的目的，但 inbox 卡片**尚非真实知识**，真实提炼属 P2。建议在 spec 显式标注，避免后续误判为已具备提炼能力。
- `better-sqlite3` 原生模块在本机仍反复出现 ABI 失配（compiled 131 / requires 127），需 `npm rebuild`。`postinstall` 已能在 `npm install` 时修复，但 Node 版本漂移后仍需重跑；`npm test` 同样依赖当前 ABI 正确。

### 8.4 P1 收口复评结论

P1 主闭环（导入→归档→标准化→SQLite→待确认卡片→DailyRun→日志）**现已可一键运行、幂等、并有测试兜底**，第 4、7 节阻断项基本清零，方向正确、质量明显提升。但"从原始归档重建"这一 P1 验收项**虽可运行却不等价**（问题 13），且被弱断言的测试掩盖（问题 14）。建议：进入 P2 前先修问题 13/14（稳定 ID 去掉 raw_path 种子 + 等价性断言），问题 8/15 一并清理；之后即可放行 P2 最小真实提炼实验。
