## 1. 规划优先级

当前项目已决定向可安装、可上线、后期可商业出售的桌面产品演进。

执行优先级：

1. `spec/产品化开发步骤.md` 是后续主规划。
2. `spec/任务拆分.md` 是概念验证、原子化规则和最小闭环约束。
3. `spec/P0产品决策.md` 是进入开发前必须确认的决策入口。
4. `spec/监督问题记录.md` 记录偏移风险，不替代任务定义。
5. `spec/执行日志.md` 记录每次实际执行结果。

如果两份规划冲突，以 `spec/产品化开发步骤.md` 为主，但不能违反 `spec/任务拆分.md` 中的核心约束：本地优先、可追溯、人工确认、不自动写入正式知识库。

## 2. 产品名称

内部项目名：AI 对话知识沉淀系统

商业名称：待定

第一版不因为命名未定而阻塞核心闭环设计。安装包、包名、官网和授权系统在 P8/P9 前再最终确认商业名称。

## 3. 目标用户

首批目标用户：

- AI 重度使用者：每天在多个 AI 产品中产生大量思考和任务记录。
- 个人创作者：需要从日常对话中沉淀观点、素材、选题和表达。
- 开发者：使用 Codex、Cursor、Claude Code 等工具推进项目，希望沉淀项目经验和决策。
- 知识工作者：需要把分散对话转成可检索、可复用的个人知识库。

暂不优先服务：

- 企业团队知识库。
- 公司内部合规归档。
- 多人协作项目管理。
- 自动公众号发布和内容分发。

## 4. 首版核心场景

首版只验证一个完整闭环：

```text
用户手动导入一个来源的 AI 对话
  -> 系统归档原始记录
  -> 转成标准化记录
  -> 生成候选知识原子
  -> 用户在待确认区审核
  -> 写入本地 Markdown 知识库
  -> 每条知识能回溯原始来源
```

本地 MVP 来源范围：

- 必须跑通：Codex 作为优先来源的完整闭环。
- 可以预留：Cursor、DeepSeek 的连接器接口和目录。
- 不要求跑通：豆包、Workbuddy。

P4 阶段再要求 Codex、Cursor、DeepSeek 三个来源分别具备稳定导入方式。

## 5. 首版不做事项

MVP 阶段不做：

- 自动网页登录和自动抓取。
- 公司项目或客户数据接入。
- 云同步。
- 团队协作。
- 商业授权实现。
- 自动更新实现。
- 官网、购买入口和账号系统。
- 向量数据库。
- 复杂图谱和高级知识分析。

商业授权、购买入口、自动更新只保留架构占位和接口边界，不在 MVP 实现。

## 6. 桌面技术栈

首版确认采用：

- 桌面壳：Electron。
- 前端：React + TypeScript。
- 本地核心：TypeScript Core Engine。
- 本地数据库：SQLite。
- 知识本体：Markdown 文件库。
- 打包：Electron Builder。

选择理由：

- Electron 对 macOS 和 Windows 支持成熟。
- Node 文件系统能力适合做本地导入、归档、索引和连接器。
- TypeScript 可以让桌面端、核心引擎和数据结构共享类型。
- 后期商业化、自动更新和安装包生态成熟。

若后续决定改用 Tauri，需要先更新本文档和 `spec/产品化开发步骤.md`。

## 7. AI 调用方式

首版采用用户自填 OpenAI-compatible API Key 的方式。

首版要求：

- 用户必须主动配置 API Key。
- 用户必须在首次 AI 提炼前确认会向模型服务发送哪些内容。
- 默认不上传完整原始记录，只发送用户选择范围内、经最小必要裁剪后的内容。
- 日志不得记录 API Key、完整原文或大段对话。
- AI 输入策略的原则和红线以 `spec/AI输入策略说明.md` 为准。
- AI 输入策略的实现细则在 P2 前再次确认。

API Key 保存策略：

- macOS 使用系统 Keychain。
- Windows 使用 Credential Manager。
- 如果系统安全凭据能力尚未实现，MVP 只能使用会话内临时 API Key，不允许落盘。
- API Key 不写入普通配置文件、日志、备份、导出文件或 Markdown 知识库。

后续再评估：

- 内置云服务。
- 本地模型。
- 团队版统一模型服务。

## 8. 本地数据存储边界

Markdown 是用户可读、可迁移的知识本体。

SQLite 是索引、运行记录、导入记录和查询加速层。

路径策略：

- 本地持久化元数据中的 `raw_path` 使用 vault-relative path。
- vault 根目录由本地设置保存，运行时再解析为绝对路径。
- AI 请求中永远不发送原始 `raw_path`。

职责划分：

- 原始记录：保存在 `raw/`，作为证据归档，不直接当知识。
- 标准化记录：写入 SQLite，同时保留原始路径和校验信息。
- 知识原子正文：写入 Markdown。
- 知识原子状态：写入 Markdown front matter，SQLite 只镜像索引。
- 每日运行记录：写入 SQLite 和 `data/daily_runs/`。
- 运行日志：写入 `logs/`，不写入敏感原文。

冲突规则：

- 知识正文和审核状态以 Markdown 为准。
- SQLite 索引可以通过 Markdown 和原始归档重建。
- Core Engine 是唯一写入入口，UI、脚本和连接器不直接绕过 Core Engine 修改正式知识。
- 用户在 Obsidian 中手动修改 Markdown 后，系统只做重新索引，不覆盖用户修改。

## 9. 隐私和安全底线

隐私约束必须前置到 P0、P1、P2。

底线规则：

- 默认不上传原始聊天记录。
- AI 调用前必须有用户授权。
- 公司项目、客户信息、无法确认权限的数据默认阻断。
- API Key 不写入普通日志和导出文件。
- 日志只记录文件名、来源、记录数量、错误类型和必要的短摘要。
- 每条知识原子必须保留来源，但证据摘录不得泄露敏感信息。

P7 会补齐完整隐私政策、删除权、导出权和配置加密，但 P1/P2 不得等到 P7 才处理最低限度隐私边界。

## 10. 数据结构版本策略

所有核心数据结构从第一版开始必须包含：

- `schema_version`
- `created_at`
- `updated_at`
- 稳定 ID

ID 规则：

- `record_id`：基于来源、会话标识、消息时间和内容校验生成。
- `atom_id`：创建时生成稳定 ID，不随标题修改变化。
- `run_id`：基于运行日期、运行开始时间和随机后缀生成。

迁移原则：

- 每次 schema 变化必须增加迁移记录。
- 旧数据不得在无备份情况下原地破坏性修改。
- SQLite 迁移和 Markdown front matter 迁移要分别设计。
- 第一版可以只预留迁移入口，但版本字段必须立即存在。
