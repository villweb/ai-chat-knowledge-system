# AI 对话知识沉淀系统产品化开发步骤

## 0. 规划优先级与执行纪律

当前项目已决定走产品化路线，后续以本文档作为主规划。

执行规则：

- `spec/产品化开发步骤.md` 是主规划。
- `spec/任务拆分.md` 是概念验证、原子化规则和最小闭环约束。
- `spec/P0产品决策.md` 是进入正式开发前的决策承载文件。
- P0 已在 2026-06-18 被用户确认。
- 进入 P1 前先初始化项目目录 git 仓库。
- MVP 阶段只要求 Codex 一个来源完整跑通，Cursor 和 DeepSeek 到 P4 再完成稳定导入。
- 商业授权、购买入口、自动更新和官网能力在 MVP 阶段只保留接口边界，不实现。

## 1. 产品化目标

把“每日扫描 AI 对话并沉淀个人知识”的能力，做成可以长期维护、可安装、可上线、后期可商业出售的桌面产品。

产品第一原则：

- 本地优先：用户的原始聊天记录、知识库、索引和运行日志默认保存在本机。
- 授权采集：只读取用户明确选择或授权的路径、导出文件和数据源。
- 人工确认：AI 只生成候选知识，正式知识必须由用户确认。
- 可追溯：每条知识原子必须能回到原始对话或导入文件。
- 跨平台：核心能力同时支持 macOS 和 Windows。
- 可产品化：安装、更新、授权、错误恢复、隐私说明和用户设置都进入正式开发范围。

## 2. 推荐产品架构

### 2.1 桌面端架构

已确认方向：

- 桌面壳：Electron。
- 前端界面：TypeScript + React。
- 本地数据：SQLite + Markdown 文件库。
- AI 流程：独立 Core Engine，提供结构化输入输出。
- 连接器：插件式 Connector，每个平台一个适配器。
- 安装包：macOS 提供 `.dmg` 或 `.pkg`，Windows 提供 `.exe` 或 `.msi`。

Tauri 只作为未来重大技术变更备选；如需切换，必须先更新 `spec/P0产品决策.md`。

存储边界以 `spec/P0产品决策.md` 为准：Markdown 是知识本体，SQLite 是标准化记录、导入记录、运行记录、知识索引和查询加速层。Core Engine 是唯一写入入口。

### 2.2 核心模块边界

```text
Desktop App
  Onboarding
  Settings
  Import Center
  Daily Review
  Knowledge Inbox
  Source Manager
  License and Update Placeholder

Core Engine
  Source Connector
  Normalizer
  Privacy Classifier
  Atom Extractor
  Deduplicator
  Knowledge Writer
  Scheduler

Local Storage
  Raw Archive
  SQLite Index
  Markdown Knowledge Vault
  Logs
  Settings
```

### 2.3 数据流

```text
用户授权来源
  -> 原始记录归档
  -> 标准化对话记录
  -> 隐私和范围判断
  -> AI 调用授权确认
  -> AI 提炼候选知识原子
  -> 去重和合并建议
  -> 待确认区
  -> 用户确认
  -> 正式知识库
```

## 3. 产品级目录规划

现有目录保留为概念验证项目目录。产品化开发时建议演进为：

```text
AI对话知识沉淀系统/
  spec/
    任务拆分.md
    产品化开发步骤.md
  app/
    desktop/
    core/
    connectors/
    storage/
    workflow/
    schemas/
    services/
  prompts/
  raw/
  data/
  knowledge/
  scripts/
  build/
    mac/
    windows/
  logs/
```

目录职责：

- `app/desktop/`：桌面端 UI、路由、状态管理、用户操作界面。
- `app/core/`：标准化、提炼、去重、写入、任务调度等核心业务。
- `app/connectors/`：Codex、Cursor、DeepSeek、豆包、Workbuddy 等来源适配器。
- `app/storage/`：SQLite、Markdown、文件索引、配置读写。
- `app/workflow/`：每日沉淀流水线和后台任务编排。
- `app/schemas/`：标准化记录、知识原子、运行结果、用户设置等数据结构。
- `app/services/`：AI 请求、授权、更新、日志、错误上报等服务。
- `build/`：安装包、签名、发布配置。

注意：该目录是产品化目标结构。P1 阶段只创建 `app/core/`、`app/schemas/`、`app/storage/`、`app/connectors/`、`app/workflow/` 等与核心引擎直接相关的最小目录；`app/desktop/` 和 `build/` 等桌面 UI、安装包、签名、发布相关目录等到对应阶段再创建。

## 5. 首个可开发版本范围

第一轮开发不要直接做完整商业版，建议先做“产品架构正确的本地 MVP”。

必须包含：

- 本地桌面端。
- 手动导入。
- Codex 一个来源完整跑通。
- Cursor 和 DeepSeek 只要求接口预留或格式调研。
- AI 提炼候选知识。
- 待确认区。
- Markdown 知识库写入。
- 本地日志。
- 基础设置。

暂缓包含：

- 自动网页抓取。
- 商业授权。
- 自动更新。
- 官网和购买入口。
- 豆包和 Workbuddy 深度接入。
- 公司项目隔离库。
- 云同步。
- 团队协作。

## 6. 开发顺序建议

1. P0：先确定桌面技术栈和本地数据模型。
2. P1：P0 被用户确认后，实现 Core Engine，不依赖 UI。
3. P2：P1 完成后，实现 AI 提炼质量控制。
4. P3：P2 完成后，做桌面端最小可用界面。
